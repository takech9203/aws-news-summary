Amazon Bedrock

Automated Reasoning ポリシーのソースドキュメント参照機能

Fidelity Report でポリシーの品質を定量評価し、ソースドキュメントまでルールを追跡可能に

2026 年 2 月 23 日

📋 概要

Amazon Bedrock の Automated Reasoning ポリシーにソースドキュメント参照機能が追加されました。ユーザーがアップロードした知識ドキュメントから形式論理のルールと変数が自動抽出され、各ルールと変数を元のドキュメントの具体的な記述にリンクして確認できるようになりました。Fidelity Report によりカバレッジスコアと精度スコアでポリシー品質を定量的に評価でき、形式論理の知識がない非技術者でもポリシーを検証・改善できます。AI ハルシネーションの検出において数学的に証明可能な保証を提供し、正しい LLM レスポンスの検出で最大 99% の精度を実現します。

Before - アップデート前の課題
  • ポリシールールが元のドキュメントのどの記述に基づいているか確認が困難
  • ポリシーの正確性検証には形式論理の知識が必要で非技術者にとって敷居が高い
  • ルールや変数の修正時に元のドキュメントとの対応関係を手動で確認
  • ポリシーのカバレッジと精度を定量的に評価する手段が限定的
After - アップデート後の改善
  • 各ルールと変数がソースドキュメントの具体的な記述にリンクされ根拠を即座に確認
  • Fidelity Report でカバレッジスコアと精度スコアによる定量的品質評価
  • 非技術者のドメインエキスパートが形式論理を理解せずにポリシーを検証・改善
  • アトミックステートメント単位でルール・変数への対応を可視化

ソースドキュメントのアップロードから FM 出力の検証までの全体フロー。Fidelity Report とソースドキュメント参照によりポリシー品質を管理。

⭐ 主要機能

📊
Fidelity Report の自動生成
ソースドキュメントからポリシーを作成する際に自動生成。カバレッジスコアでドキュメントの反映度を、精度スコアでルールの正確性を 0.0 - 1.0 のスケールで定量評価します。
📏
ルールレポート
各ルールに対してソースドキュメントの具体的なグラウンディングステートメントを特定。ルールを正当化するグラウンディングジャスティフィケーションと個別の精度スコアを提供します。
🔤
変数レポート
各変数の定義を裏付けるソースドキュメントの記述を特定。根拠の説明と個別の精度スコアにより、変数定義の正確性を確認できます。
📌
ドキュメントソース
ソースドキュメントをアトミックステートメント (個別の不可分な事実) に分解し、行番号でアノテーション。元のドキュメントの正確な位置を追跡できます。

Fidelity Report のフィールド

フィールド 型 説明

検証結果の種類

結果 説明
VALID レスポンスがポリシールールに準拠
INVALID レスポンスがポリシールールに違反
SATISFIABLE レスポンスがルールと矛盾しないが、完全な検証ではない
IMPOSSIBLE 入力条件がポリシーの公理と矛盾
TRANSLATION_AMBIGUOUS 自然言語から形式論理への変換が曖昧
TOO_COMPLEX 検証の複雑性が上限を超過

bedrock = boto3.client('bedrock', region_name='us-east-1')

# ポリシーバージョンを作成してデプロイ
response = bedrock.create_guardrail(
    name='my-guardrail',
    automatedReasoningPolicyConfig={
        'policyArn': 'arn:aws:bedrock:us-east-1:123456789012:automated-reasoning-policy/my-policy',
        'policyVersion': '1'
    }

👍 メリット

💼 ビジネス面

  • ポリシー品質の可視化: カバレッジスコアと精度スコアで品質を定量的に把握
  • ドメインエキスパートの活用: 形式論理の知識なしにソースドキュメント参照でポリシーを検証・改善
  • コンプライアンスの強化: FM 生成コンテンツが最大 99% の精度でポリシーに準拠していることを検証
  • AI ハルシネーション対策: 数学的に証明可能な保証で AI の誤った回答を高精度で検出

🛠️ 技術面

  • トレーサビリティ: 各ルールと変数をソースドキュメントの具体的な記述まで追跡
  • アトミックステートメント分解: ドキュメントを不可分な事実に分解し行番号でアノテーション
  • 反復的改善: Fidelity Report のフィードバックに基づきポリシーを段階的に改善
  • 形式検証: SMT-LIB 構文による数学的に厳密なルール検証

⚠️ 制約事項・注意点

制限事項
  • テスト生成は限定されたリージョンでのみ利用可能
  • ソースドキュメントにはサイズ制限がある
  • 各ポリシーは特定のドメインに焦点を当てるべきであり、複数の無関係な領域をカバーするポリシーは推奨されない
  • 自然言語から形式論理への変換には変換エラーが含まれる可能性がある
考慮すべき点
  • 変数の説明文が変換精度に最も大きく影響するため、適切な説明を記述することが重要
  • ベアアサーション (if-then 構造を持たないルール) は公理を作成するため、意図しない IMPOSSIBLE 結果を引き起こす可能性がある
  • 複雑なドメインでは反復的なポリシー構築が推奨される

💡 ユースケース

👥
人事ポリシーのコンプライアンス検証
企業の人事部門が、社員向け AI チャットボットの回答が人事ポリシー (休暇制度、福利厚生、就業規則など) に準拠しているかを検証する。カバレッジスコアで全ルールのカバー状況を、精度スコアで変換の正確性を確認。
人事担当者が形式論理を理解せずに、ソースドキュメント参照を通じてポリシーの正確性を検証。AI チャットボットの回答が人事ポリシーに準拠していることを数学的に保証。
🏦
金融規制ガイドラインの準拠検証
金融機関が、ローン審査や投資アドバイスを行う AI アシスタントの回答が金融規制ガイドラインに準拠しているかを検証する。元のガイドラインのどの条項に基づくルールかをソースドキュメント参照で確認。
規制対応チームがポリシーの忠実度を定量的に評価し、コンプライアンスリスクを低減。監査時にルールの根拠をソースドキュメントまで追跡可能。
📦
製品仕様に基づく顧客サポート
製品の仕様書や返品ポリシーをソースドキュメントとして使用し、顧客サポート AI の回答が正確な製品情報に基づいているかを検証する。アトミックステートメントの一覧で仕様書の網羅性を評価。
製品仕様の変更時にポリシーの更新箇所をソースドキュメント参照から特定。AI ハルシネーションによる誤った製品情報の提供を防止。
