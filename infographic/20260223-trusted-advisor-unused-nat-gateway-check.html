AWS Trusted Advisor

Compute Optimizer を活用した未使用 NAT Gateway チェックの強化

高精度な検出で誤検知を削減し、コスト最適化レコメンデーションを強化

2026 年 2 月 23 日

📋 概要

AWS Trusted Advisor が、AWS Compute Optimizer を活用した未使用 NAT Gateway の検出機能を強化しました。新しい検出方法では、32 日間のルックバック期間にわたって複数の CloudWatch メトリクスを分析し、NAT Gateway がルートテーブルに関連付けられているかを検証します。これにより、重要なバックアップリソースの誤検知を大幅に削減し、各レコメンデーションには推定月額コスト削減額が含まれるようになりました。

Before - Inactive NAT Gateways
  • 30 日間のデータ処理量のみで判定し、検出精度が限定的
  • ルートテーブルとの関連付けを確認せず、バックアップ用途の NAT Gateway も誤検知
  • コスト削減額の見積もりがなく、優先順位付けが困難
After - Idle NAT gateways
  • 32 日間にわたり複数の CloudWatch メトリクスを包括的に分析
  • ルートテーブル関連付けを検証し、バックアップリソースの誤検知を回避
  • 推定月額コスト削減額を表示し、優先順位付けが容易
  • Cost Optimization Hub との統合で組織全体を一元管理

Compute Optimizer が CloudWatch メトリクスの分析とルートテーブルの検証を行い、バックアップリソースを除外したうえで、推定コスト削減額を含むレコメンデーションを配信

⭐ 主要機能

📊
Compute Optimizer を活用した高精度な検出
32 日間のルックバック期間で CloudWatch メトリクスを分析。アクティブ接続数、VPC からの受信パケット数、宛先からの受信パケット数を評価し、従来の 30 日間のデータ処理量のみの判定から大幅に精度が向上。
🔌
ルートテーブル関連付けの検証
NAT Gateway が AWS ルートテーブルに関連付けられているかを確認。ディザスタリカバリや高可用性のためのバックアップリソースを保護し、誤検知を大幅に削減。
💰
コスト削減額の見積もり
各レコメンデーションに推定月額コスト削減額を表示。対応の優先順位付けが金額ベースで可能になり、Cost Optimization Hub で組織全体の重複排除された削減額を確認可能。
✅
Trusted Advisor との統合
Trusted Advisor コンソールで直接レコメンデーションを確認可能。API を通じたプログラマティックなアクセスにも対応し、新しい statusReason 属性で詳細な理由を提供。

⚙️ 設定方法

前提条件

  • AWS Business Support、Enterprise Support、または Unified Operations プランに加入していること
  • Cost Optimization Hub へのオプトインが完了していること
  • AWS Compute Optimizer へのオプトインが完了していること
  • 適切な IAM 権限が設定されていること

ステップ 1: Compute Optimizer へのオプトイン

Bash Compute Optimizer へのオプトイン 
# Compute Optimizer へのオプトイン
aws compute-optimizer update-enrollment-status \
  --status Active

オプトイン後、24 時間以内に未使用 NAT Gateway のレコメンデーションが生成されます。

ステップ 2: Cost Optimization Hub へのオプトイン

💻 AWS Billing
コンソールを開く
→
📈 Cost Optimization
Hub を選択
→
✅ Opt in を選択
して有効化

ステップ 3: レコメンデーション確認

Bash Trusted Advisor でコスト最適化チェックの結果を取得 
# Trusted Advisor でコスト最適化チェックの結果を取得
aws trustedadvisor list-recommendations \
  --pillar CostOptimizing \
  --source CostOptimizationHub

ステップ 4: 個別リソースの詳細確認

Bash Compute Optimizer で未使用 NAT Gateway の詳細を確認 
# Compute Optimizer で未使用 NAT Gateway の詳細を確認
aws compute-optimizer get-idle-recommendations \
  --resource-type NatGateway

ステップ 5: 未使用 NAT Gateway の削除

Bash 未使用と確認された NAT Gateway を削除 
# 未使用と確認された NAT Gateway を削除
aws ec2 delete-nat-gateway \
  --nat-gateway-id nat-xxxxxxxxxxxxxxxxx

# 関連する Elastic IP の解放
aws ec2 release-address \
  --allocation-id eipalloc-xxxxxxxxxxxxxxxxx

削除前に、ディザスタリカバリや高可用性の要件を考慮し、関連する Elastic IP アドレスの解放も検討してください。

👍 メリット

💼 ビジネス面

  • コスト最適化の精度向上: Compute Optimizer の高度な分析により、本当に不要なリソースのみを特定し、安全にコスト削減を実現
  • 優先順位付けの容易化: 推定月額コスト削減額が表示されるため、最もインパクトの大きいリソースから対応可能
  • 組織全体の可視化: Cost Optimization Hub との統合により、マルチアカウント環境全体のコスト最適化機会を一元的に把握

🛠️ 技術面

  • 誤検知の削減: ルートテーブル関連付けの検証により、バックアップ用途のリソースを誤って削除対象にするリスクを最小化
  • 包括的なメトリクス分析: 複数の CloudWatch メトリクスを 32 日間にわたって分析し、より正確な未使用判定を実現
  • プログラマティックなアクセス: Trusted Advisor API および Compute Optimizer API を通じて自動化ワークフローに統合可能

⚠️ 制約事項・注意点

制限事項
  • Cost Optimization Hub と Compute Optimizer の両方へのオプトインが必須
  • AWS Business Support 以上のサポートプランが必要
  • レコメンデーション生成まで最大 24 時間かかる
  • AWS GovCloud (US) リージョンおよび中国リージョンでは利用不可
考慮すべき点
  • バックアップリソースの自動除外は完全ではない可能性がある。高可用性やディザスタリカバリのアーキテクチャを手動で確認してから削除すること
  • NAT Gateway を削除すると、関連するプライベートサブネットからインターネットへのアウトバウンド通信が遮断される
  • NAT Gateway に関連付けられた Elastic IP アドレスは自動的には解放されない。別途対応が必要

💡 ユースケース

🚀
ワークロード移行後のリソースクリーンアップ
オンプレミスから AWS への移行プロジェクトが完了し、移行期間中に使用していた一時的な VPC やサブネットに NAT Gateway が残存している。
移行完了後に不要になった NAT Gateway を確実に特定し、月額コストを削減。ルートテーブル検証により、まだ使用中のリソースを誤って削除するリスクを回避。
🏢
マルチアカウント環境での定期コスト監査
大規模な AWS Organizations 環境で、複数のアカウントにまたがる NAT Gateway のコスト最適化を定期的に実施したい。
Cost Optimization Hub の重複排除機能により、組織全体の正確なコスト削減額を把握。定期的な監査プロセスに組み込むことで、未使用リソースの蓄積を防止。
💻
開発・テスト環境のコスト管理
開発・テスト環境で一時的に作成された VPC の NAT Gateway が、テスト完了後もそのまま残っている。
開発・テスト環境で放置されがちな NAT Gateway を自動検出し、不要なコストを排除。NAT Gateway 1 つあたり月額約 $32 のコスト削減が可能。
