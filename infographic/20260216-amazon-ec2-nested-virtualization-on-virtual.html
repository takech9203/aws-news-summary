ネステッド仮想化サポート

2026 年 2 月 16 日

概要

仮想 Amazon EC2 インスタンス上でネステッド仮想化が利用可能になりました。これまでベアメタル EC2 インスタンスでのみサポートされていたネステッド仮想化が、C8i、M8i、R8i の仮想インスタンスでも利用できるようになり、KVM や Hyper-V などのハイパーバイザーを実行してネストされた仮想マシンを作成・管理できます。

Nitro System がプロセッサ拡張機能 (Intel VT-x) を仮想インスタンスに渡すことで実現され、L0 (Nitro ハイパーバイザー) / L1 (EC2 インスタンス) / L2 (ネスト VM) の 3 層アーキテクチャで構成されます。すべての商用リージョンで追加料金なしで利用可能です。

Before - アップデート前の課題
  • ネステッド仮想化にはベアメタル EC2 インスタンスが必要で、コストが高くサイズの柔軟性に欠けていた
  • ベアメタルインスタンスは大型サイズのみで、小規模な開発・テスト用途には過剰なリソースだった
  • Docker Desktop、WSL2、Android Studio エミュレータなどの開発ツールをクラウド上で実行する選択肢が限定的だった
After - アップデート後の改善
  • 仮想 EC2 インスタンス上で KVM や Hyper-V を実行し、ネストされた仮想マシンを作成可能に
  • ベアメタル不要で、幅広いインスタンスサイズから要件に合ったものを選択可能に
  • Docker Desktop、WSL2、Android Studio エミュレータ、QEMU を柔軟に利用可能に
  • 追加料金なしで利用可能なため、コスト効率が大幅に改善

主要機能

💻
仮想インスタンスでのネステッド仮想化
仮想 EC2 インスタンス上で KVM および Hyper-V ハイパーバイザーを実行可能。Nitro System が Intel VT-x をパススルーすることで L0 / L1 / L2 の 3 層構成を実現。
🚀
新規インスタンス起動時の有効化
起動ウィザードの「Advanced details」で「Nested virtualization」を「Enable」に設定。AWS CLI では --cpu-options オプションで指定。
🔄
既存インスタンスへの適用
停止状態のインスタンスに対して、コンソールまたは modify-instance-cpu-options コマンドでネステッド仮想化を有効化可能。移行不要。
💰
追加料金なし
ネステッド仮想化の利用に追加コストは発生しない。通常の EC2 インスタンス料金のみで機能を活用可能。

設定方法

前提条件
  • AWS アカウントと EC2 インスタンスを起動するための適切な IAM 権限
  • C8i、M8i、または R8i インスタンスタイプが利用可能なリージョン
  • 既存インスタンスに適用する場合は、インスタンスが停止状態であること
ステップ 1: 新規インスタンスの起動時にネステッド仮想化を有効化
Bash C8i インスタンスでネステッド仮想化を有効にして起動 
aws ec2 run-instances \
    --image-id ami-0abcdef1234567890 \
    --instance-type c8i.4xlarge \
    --cpu-options "NestedVirtualization=enabled" \
    --key-name my-key-pair

--cpu-options パラメータで NestedVirtualization=enabled を指定することで、インスタンス内でハイパーバイザーを実行できるようになります。

ステップ 2: 既存インスタンスでネステッド仮想化を有効化
Bash 既存インスタンスへの適用 
# まずインスタンスを停止
aws ec2 stop-instances --instance-id i-1234567890abcdef0

# 停止を確認後、ネステッド仮想化を有効化
aws ec2 modify-instance-cpu-options \
    --instance-id i-1234567890abcdef0 \
    --core-count 4 \
    --threads-per-core 2 \
    --nested-virtualization enabled

# インスタンスを再起動
aws ec2 start-instances --instance-id i-1234567890abcdef0

既存インスタンスでは停止状態でのみ設定変更が可能です。CPU オプションを変更してネステッド仮想化を有効化した後、インスタンスを再起動します。

ステップ 3: コンソールからの設定

新規インスタンス: インスタンス起動ウィザードで「Advanced details」を展開し、「Nested virtualization」で「Enable」を選択

既存インスタンス: インスタンスを選択 → 「Actions」→「Instance settings」→「Change CPU options」→「Nested virtualization」で「Enable」を選択

ユースケース

📱 ユースケース 1: モバイルアプリケーション開発とテスト
シナリオ: モバイルアプリ開発チームが、Android Studio のエミュレータを使用してクラウド上で自動テストパイプラインを構築したい
Bash M8i インスタンスで Android エミュレータ環境を起動 
aws ec2 run-instances \
    --image-id ami-0abcdef1234567890 \
    --instance-type m8i.2xlarge \
    --cpu-options "NestedVirtualization=enabled" \
    --key-name dev-key-pair \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Purpose,Value=android-emulator}]'
✔ ベアメタルインスタンスと比較してコストを大幅に削減しつつ、Android エミュレータをハードウェアアクセラレーション付きで実行でき、テストの速度と効率が向上
🚗 ユースケース 2: 自動車の車載ハードウェアシミュレーション
シナリオ: 自動車メーカーが、車載 ECU のソフトウェアをクラウド上でシミュレーションしてテストしたい
Bash C8i インスタンスでコンピューティング最適化環境を構築 
aws ec2 run-instances \
    --image-id ami-0abcdef1234567890 \
    --instance-type c8i.8xlarge \
    --cpu-options "NestedVirtualization=enabled" \
    --key-name simulation-key \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Purpose,Value=vehicle-simulation}]'
✔ 専用の物理ハードウェアを用意することなく、クラウド上で車載環境をシミュレーションでき、開発サイクルの短縮とインフラコストの削減を実現
💻 ユースケース 3: Windows ワークステーションでの WSL2 利用
シナリオ: 開発チームが、クラウド上の Windows ワークステーションで WSL2 を使用して Linux 開発環境と Windows 開発環境を同時に利用したい
Bash R8i インスタンスでメモリ最適化 Windows ワークステーションを起動 
aws ec2 run-instances \
    --image-id ami-windows-server-2022 \
    --instance-type r8i.4xlarge \
    --cpu-options "NestedVirtualization=enabled" \
    --key-name windows-key \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Purpose,Value=wsl2-workstation}]'
✔ 1 つの Windows インスタンス上で WSL2 を通じて Linux 環境を利用でき、クロスプラットフォーム開発の生産性が向上。メモリ最適化の R8i インスタンスにより、複数の開発環境を同時に快適に実行可能

メリット

💰 ビジネス面
  • コスト削減: ベアメタルではなく小型で安価な仮想インスタンスでネステッド仮想化を実現
  • 柔軟なリソース選択: C8i、M8i、R8i の幅広いインスタンスサイズからワークロードに最適なリソースを選択可能
  • 開発効率の向上: クラウド上で Docker Desktop、WSL2、エミュレータなどの開発ツールを手軽に利用でき、開発サイクルが短縮
⚡ 技術面
  • 幅広いインスタンスサイズ: ベアメタルに限定されず、さまざまなサイズでネステッド仮想化を利用可能
  • 追加料金なし: 既存のインスタンス料金のみで機能を活用可能
  • Nitro System による高いセキュリティ: EC2 インスタンス間の強力な分離境界を維持
  • 既存インスタンスへの適用: 新規だけでなく既存のインスタンスにも設定変更で有効化可能

制約事項・考慮点

⚠ 制限事項
  • 対応インスタンスタイプは C8i、M8i、R8i のみで、他のインスタンスファミリーでは利用不可
  • L1 ハイパーバイザーとして KVM と Hyper-V のみがサポート
  • 192 CPU を超える Windows インスタンスではネステッド仮想化は非対応
  • Windows インスタンスでは VSM が自動的に無効化され、Credential Guard が利用不可
  • ネステッド仮想化が有効なインスタンスではハイバネーションが非対応
ℹ 考慮すべき点
  • パフォーマンスに敏感なワークロードや厳格なレイテンシ要件がある場合は、引き続きベアメタルインスタンスの利用を検討
  • インスタンス内のハイパーバイザー、ネスト VM、ゲスト OS、アプリケーション、データのセキュリティはユーザーの責任 (共有責任モデル)
  • 既存インスタンスで有効化する場合は、インスタンスの停止が必要

料金

ネステッド仮想化の利用に追加料金は発生しません。通常の EC2 インスタンス料金のみが適用されます。

料金例 - 米国東部 (バージニア北部) オンデマンド料金参考値
インスタンスタイプ vCPU メモリ (GiB) 用途
c8i.4xlarge 16 32 コンピューティング最適化ワークロード
m8i.4xlarge 16 64 汎用ワークロード
r8i.4xlarge 16 128 メモリ最適化ワークロード

ベアメタルインスタンスと比較して、同等の vCPU 数で仮想インスタンスを使用することにより、より細かい粒度でリソースを選択でき、コスト最適化が可能です。最新の料金については Amazon EC2 料金ページをご確認ください。

利用可能リージョン

ネステッド仮想化は、C8i、M8i、R8i インスタンスが利用可能なすべての商用リージョンで提供されています。
