ストレージ最適化インスタンスの一般提供開始

2026 年 2 月 26 日

概要

Amazon EC2 ストレージ最適化 I8g.metal-48xl インスタンスが一般提供 (GA) されました。AWS Graviton4 プロセッサを搭載し、前世代の I4g インスタンスと比較して最大 60% 優れたコンピューティング性能を提供します。

最新の第 3 世代 AWS Nitro SSD を使用し、TB あたり最大 65% 優れたリアルタイムストレージ性能を実現。ストレージ I/O レイテンシーを最大 50% 削減、レイテンシーのばらつきを最大 60% 削減しています。

I8g.metal-48xl は 192 vCPU、1,536 GiB メモリ、45 TB ローカル NVMe ストレージを備え、最大 100 Gbps のネットワーク帯域幅と 60 Gbps の EBS 専用帯域幅を提供する I8g ファミリー最大のインスタンスサイズです。

Before - アップデート前の課題
  • 前世代の I4g では大規模トランザクション DB やリアルタイム分析に対するコンピューティング性能が不足
  • ストレージ I/O レイテンシーとばらつきが大きく、一貫した性能確保が困難
  • I8g ファミリーで metal-48xl サイズが利用不可。ベアメタルで最大規模のリソースに対応できなかった
  • TB あたりのストレージ性能に制約があり、リアルタイムワークロードでボトルネック発生
After - アップデート後の改善
  • 192 vCPU、1,536 GiB メモリ、45 TB ローカルストレージをベアメタル環境で利用可能
  • 第 3 世代 Nitro SSD により TB あたり最大 65% 優れたリアルタイムストレージ性能
  • ストレージ I/O レイテンシーが最大 50% 低減し、DB ワークロードの応答性が向上
  • Graviton4 により前世代比で最大 60% のコンピューティング性能向上

I8g.metal-48xl は AWS Graviton4 と第 3 世代 Nitro SSD を搭載し、Nitro System によるハードウェアレベルの最適化を実現。トランザクション DB、NoSQL DB、リアルタイム分析に最適化されています。

主要機能

⚡
AWS Graviton4 プロセッサ
前世代比最大 60% のコンピューティング性能向上。2 倍のキャッシュサイズ、DDR5-5600 メモリ採用でメモリ帯域幅が改善。全高速インターフェースを完全暗号化。
💿
第 3 世代 AWS Nitro SSD
TB あたり最大 65% 優れたストレージ性能。I/O レイテンシー 50% 削減、ばらつき 60% 削減。常時暗号化によるセキュリティ確保。
🛡️
AWS Nitro System 基盤
CPU 仮想化、ストレージ、ネットワーキングを専用ハードウェアにオフロード。ベアメタルインスタンスとして仮想化オーバーヘッドなしで利用可能。
💻
大規模リソース構成
192 vCPU、1,536 GiB メモリ、45 TB ローカル NVMe ストレージ。100 Gbps ネットワーク、60 Gbps EBS 専用帯域。

設定方法

前提条件
  • AWS アカウントが有効化されている
  • 利用可能なリージョンでリソースを作成する
  • 適切な IAM 権限が設定されている
  • I8g.metal-48xl のサービスクォータが十分に確保されている
ステップ 1: I8g.metal-48xl インスタンスの起動
Bash AWS CLI で I8g.metal-48xl インスタンスを起動 
aws ec2 run-instances \
  --instance-type i8g.metal-48xl \
  --image-id ami-xxxxxxxxxxxxxxxxx \
  --key-name my-key-pair \
  --security-group-ids sg-xxxxxxxxxxxxxxxxx \
  --subnet-id subnet-xxxxxxxxxxxxxxxxx \
  --block-device-mappings '[{"DeviceName":"/dev/xvda","Ebs":{"VolumeSize":100,"VolumeType":"gp3"}}]' \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=i8g-metal-48xl-db}]'

指定した AMI、キーペア、セキュリティグループ、サブネットを使用して I8g.metal-48xl インスタンスを起動します。ベアメタルインスタンスのため、完全なハードウェアアクセスが可能です。

ステップ 2: ローカル NVMe ストレージの確認と設定
Bash NVMe ストレージのセットアップとベンチマーク 
# NVMe デバイスを確認
lsblk

# NVMe ストレージをフォーマットしてマウント
sudo mkfs.xfs /dev/nvme1n1
sudo mkdir -p /data
sudo mount /dev/nvme1n1 /data

# ストレージ性能の確認
sudo fio --name=randread --ioengine=libaio --iodepth=32 \
  --rw=randread --bs=4k --direct=1 --size=1G \
  --numjobs=4 --runtime=60 --filename=/data/testfile

12 本の NVMe ドライブ (各 3,750 GB) にアクセスできます。フォーマット、マウント後に fio で I/O 性能のベンチマークを実行します。

ステップ 3: データベースワークロードの設定
Bash PostgreSQL の設定例 - ローカル NVMe ストレージ活用 
# PostgreSQL データディレクトリをローカル NVMe に配置
sudo mkdir -p /data/postgresql
sudo chown postgres:postgres /data/postgresql
sudo -u postgres initdb -D /data/postgresql/data

# postgresql.conf の設定最適化
sudo -u postgres cat >> /data/postgresql/data/postgresql.conf << 'EOF'
shared_buffers = 384GB
effective_cache_size = 1152GB
work_mem = 2GB
maintenance_work_mem = 8GB
max_connections = 500
EOF

# PostgreSQL の起動
sudo -u postgres pg_ctl -D /data/postgresql/data start

1,536 GiB のメモリとローカル NVMe ストレージを活用した PostgreSQL 最適化設定例です。shared_buffers をメモリの約 25%、effective_cache_size を約 75% に設定しています。

ユースケース

💳 ユースケース 1: 大規模トランザクションデータベース
シナリオ: 金融機関がリアルタイム取引処理システムの PostgreSQL データベースを I8g.metal-48xl 上で運用し、低レイテンシーかつ高スループットのトランザクション処理を実現
Bash RAID 0 構成で NVMe ストレージを最大限に活用 
sudo mdadm --create /dev/md0 --level=0 --raid-devices=12 \
  /dev/nvme1n1 /dev/nvme2n1 /dev/nvme3n1 /dev/nvme4n1 \
  /dev/nvme5n1 /dev/nvme6n1 /dev/nvme7n1 /dev/nvme8n1 \
  /dev/nvme9n1 /dev/nvme10n1 /dev/nvme11n1 /dev/nvme12n1
sudo mkfs.xfs /dev/md0
sudo mount /dev/md0 /data
✔ 12 本の NVMe を RAID 0 で構成し最大 I/O スループットを確保。1,536 GiB メモリで DB の大部分をキャッシュ可能。前世代比 60% の性能向上と 50% のレイテンシー削減でトランザクション処理が大幅に向上。
🛒 ユースケース 2: NoSQL データベースクラスター
シナリオ: EC サイトが MongoDB クラスターを I8g 上で運用し、商品カタログ、ユーザーセッション、注文履歴などの大規模データセットに対するリアルタイムクエリを処理
YAML MongoDB の WiredTiger ストレージエンジン設定 
storage:
  dbPath: /data/mongodb
  engine: wiredTiger
  wiredTiger:
    engineConfig:
      cacheSizeGB: 1024
    collectionConfig:
      blockCompressor: snappy
net:
  port: 27017
  bindIp: 0.0.0.0
replication:
  replSetName: rs0
✔ 第 3 世代 Nitro SSD の低レイテンシー特性により WiredTiger の読み取り性能が向上。大容量メモリをキャッシュに割り当て、頻繁にアクセスされるデータをメモリ内に保持し応答時間を短縮。
📈 ユースケース 3: リアルタイム分析基盤
シナリオ: データ分析企業が Apache Spark クラスターを I8g インスタンス群で構成し、ストリーミングデータに対するリアルタイム分析と機械学習モデルの推論を同時実行
Bash Spark 設定 - I8g.metal-48xl のリソースを最大限に活用 
cat > spark-defaults.conf << 'EOF'
spark.executor.memory=128g
spark.executor.cores=16
spark.driver.memory=64g
spark.local.dir=/data/spark-local
spark.sql.shuffle.partitions=384
spark.executor.instances=10
EOF
✔ 45 TB のローカル NVMe をシャッフルデータやキャッシュに使用。1,536 GiB メモリで大規模データセットをインメモリ処理。I/O レイテンシーばらつき 60% 削減で Spark ジョブの実行時間が予測可能に。

メリット

💰 ビジネス面
  • Graviton4 のエネルギー効率改善と TB あたりのストレージ性能向上により、同等のワークロードをより少ないインスタンスで処理可能
  • ストレージ I/O レイテンシー 50% 削減とばらつき 60% 削減により、エンドユーザー体験が向上
  • オンデマンド、Savings Plans、スポット、専用インスタンス、Dedicated Hosts に対応し、コスト最適化の選択肢が豊富
⚡ 技術面
  • ベアメタルアクセスで仮想化オーバーヘッドなし。カスタムハイパーバイザーやコンテナランタイムの実行が可能
  • 第 3 世代 Nitro SSD によるレイテンシーばらつきの大幅削減で、予測可能な性能を提供
  • Amazon Linux、Ubuntu、RHEL、SUSE、Debian、FreeBSD など主要な Linux ディストリビューションに対応
  • ARM アーキテクチャ向けに最適化された広範な Graviton エコシステムを活用可能

制約事項・考慮点

⚠ 制限事項
  • ベアメタルインスタンスのため、起動時間が仮想化インスタンスと比較して長くなる場合がある
  • ローカル NVMe ストレージはインスタンスストアのため、停止や終了時にデータが消失する。永続化には EBS や S3 へのバックアップが必須
  • ARM アーキテクチャ (Graviton4) のため、x86 専用のソフトウェアは直接実行できない
ℹ 考慮すべき点
  • I4g からの移行は比較的容易だが、アプリケーションが ARM アーキテクチャに対応しているか事前確認が必要
  • metal-48xl は大規模インスタンスのため、小規模ワークロードには I8g の他サイズの方がコスト効率に優れる場合がある
  • ストレージ集約型ワークロードのネットワーク使用パターンはバースト型が多く、バースト可能なネットワーク帯域幅の仕様をサイズごとに確認することを推奨
